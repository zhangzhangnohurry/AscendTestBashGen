## Scope

This document is a command reference for `npu-smi` on Atlas 350 acceleration cards. It covers device information query, setup operations, firmware upgrade, clearing device information, and TLS certificate management. It is directly useful for generating device-shell validation scripts, diagnostics, and maintenance command plans.

## Key Chapters

- `j003` Ascend HDK接口和命令工具简介: defines `npu-smi`, DCMI, `hccn_tool`, and deployment scenario categories.
- `j004` npu-smi工具使用导读: global usage constraints and 950-generation parameter change.
- `j005`/`j006`: top-level help and version queries.

## Global Rules for Script Generation

- Do not run `npu-smi` concurrently from multiple threads. The guide explicitly says multi-thread concurrent use is unsupported.
- Treat command output shown in the document as examples only; scripts should parse actual device output defensively.
- Unless a chapter says otherwise, container scenarios mean ordinary containers.
- For Ascend 950 generation, use `-i dev_id`; the older `-i id -c chip_id` format is no longer compatible. For generations before 950, continue using `-i id -c chip_id`.
- `dev_id` is the software logical device ID, corresponding to `/dev/davinciX`; obtain it from `npu-smi info -l`.
- Many commands include support tables by deployment scenario and user type. The common headings are physical machine/bare-metal, physical machine plus privileged container, physical machine plus ordinary container, root user, and runtime user group. Treat unsupported (`N`) or not-applicable (`NA`) cells as hard constraints when generating scripts for a known deployment.

### Information Query: `npu-smi info`

| Purpose | Command template | Notes |
| --- | --- | --- |
| Basic info for all devices | `npu-smi info` | If no process is running, output may include `No running processes found in NPU dev_id`. In host/privileged-container scenarios it can show processes from host, ordinary containers, and privileged containers; in ordinary containers it only shows ordinary-container processes. |
| Device mapping | `npu-smi info -m` | Maps NPU ID, Slot ID, Chip ID, physical ID, and chip name. |
| List all NPU devices | `npu-smi info -l` | Use this before commands that require `dev_id`. |
| Board details | `npu-smi info -t board -i dev_id` | Non-root users may see `NA` for firmware version and `NOT SUPPORT` for driver/firmware compatibility. |
| Common device info | `npu-smi info -t common -i dev_id` | Common health/temperature/basic fields. |
| Flash info | `npu-smi info -t flash -i dev_id` | Device flash information. |
| Memory info | `npu-smi info -t memory -i dev_id` | Includes on-chip memory temperature. |
| Monitoring data | `npu-smi info watch` | Query all devices. |
| Monitoring data for one device | `npu-smi info watch -i dev_id` | Query specified device. |
| Monitoring data with interval/type | `npu-smi info watch -i dev_id -d delay_seconds -s watch_type` | `delay_seconds` range is `1..100`, default `1`. `watch_type` can combine types; `t` means temperature. When profiling is enabled, AI Core `%` is unsupported and returns `0`, which is not meaningful. |
| Usage statistics | `npu-smi info -t usages -i dev_id` | When profiling is enabled, Aicore/Aivector usage rates are unsupported and return `0`, which is not meaningful. |
| Temperature | `npu-smi info -t temp -i dev_id` | Temperature in Celsius. |
| Power | `npu-smi info -t power -i dev_id` | Device power. |
| Voltage | `npu-smi info -t volt -i dev_id` | Device voltage. |
| Health | `npu-smi info -t health -i dev_id` | If multiple alarms exist, the most severe alarm is reported as the health state. `UNKNOWN` means the device does not exist or has not started. |
| Current fault events | `npu-smi info -t current-fault-event -i dev_id` | Fault-event output should be resolved using the referenced Ascend 950 black-box error-code or health-management fault docs. |
| ECC error count | `npu-smi info -t ecc -i dev_id` | Pending-isolation memory page count refreshes to isolated page count after NPU reset, OS reboot, iBMC power cycle, or AC power cycle. |
| ECC enable status | `npu-smi info -t ecc-enable -i dev_id` | HBM ECC enable status defaults to `True`. |
| Sensors | `npu-smi info -t sensors -i dev_id` | Copper optical modules may show invalid temperature as `NA`; absent optical modules omit the field. |
| System time | `npu-smi info -t sys-time -i dev_id` | Device system time. |
| KMSAgent service status | `npu-smi info -t key-manage` | Query key-management process status. |
| Process memory | `npu-smi info -t proc-mem -i dev_id` | Query device process memory usage. |
| Physical ID remap | `npu-smi info -t phyid-remap -p phy_id` | Query NPU ID, Slot ID, and Chip ID for a physical ID. |
| Topology | `npu-smi info -t topo` | Query CPU/NPU affinity and multi-NPU topology. |
| Super pod info | `npu-smi info -t spod-info -i dev_id` | Unsupported devices may return `This device does not support xxx`. |
| Die-to-die SIO status | `npu-smi info -t sio-info -i dev_id` | Query die SIO status. |

### Set Operations: `npu-smi set`

| Purpose | Command template | Rules / constraints |
| --- | --- | --- |
| Help | `npu-smi set -h` or `npu-smi set --help` | Shows set command help. |
| Reset NPU | `npu-smi set -t reset -i dev_id [-m 1]` | Without `-m 1` means out-of-band hot reset; with `-m 1` means in-band hot reset. Stop all NPU services before reset. NIC status changing UP->DOWN->UP and iBMC transient network alarms during reset are expected. |
| Clear ECC error count for a specified chip | `npu-smi set -t clear-ecc-info -i dev_id` | High risk: clears on-chip memory multi-bit ECC fault addresses pending isolation, invalidating cold isolation. Only clears historical accumulated and isolated-page statistics for the specified chip; ECC counts are also cleared automatically after device restart. |
| Set KMSAgent service state | `npu-smi set -t key-manage -s value` | Requires certificate setup first. `stop` means close service and disable autostart; it is the default. OpenSSL must be `>= 3.0.0`; use a current non-vulnerable version. |
| Set syslog persistence mode | `npu-smi set -t sys-log-enable -d mode` | `mode` is `0` or `1`, default `0`. `0` disables syslog persistence; `1` enables it. Persistent across system reboot. If config is reported as maliciously modified, run `npu-smi set -t clear-syslog-cfg` first. |
| Continuously collect all device logs | `npu-smi set -t sys-log-dump -s level -f path` | `level` is integer `1..10`; `path` must already exist, be absolute, and grant read/write/execute permission to the current user. If persistence is enabled, log collection automatically resumes after reboot. |
| Stop log process and clear config | `npu-smi set -t clear-syslog-cfg` | Ends all continuous log collection processes and clears related configuration. |

Syslog sizing rule from `j015`:

- Current timestamp log file: level 1 gives about 100 MB per NPU; each higher level adds about 100 MB per NPU.
- Historical timestamp logs: level 1 gives 200 MB per NPU; each higher level adds 200 MB per NPU; up to 99 historical timestamp files are kept.
- Total storage estimate: level 1 needs 300 MB per NPU; each higher level adds 300 MB per NPU. Scripts should compute `required_space ~= device_count * level * 300 MB`.
- During driver upgrade or overwrite install, ongoing syslog collection is closed first and automatically reopened after completion.

### Firmware Upgrade

| Purpose | Command template | Rules / constraints |
| --- | --- | --- |
| Help | `npu-smi upgrade -h` or `npu-smi upgrade --help` | Shows upgrade command help. |
| Query firmware version | `npu-smi upgrade -b item -i dev_id` | Document says only VRD firmware version query is supported, while examples include `vrd` and `vdm`; treat non-`vrd` as version-sensitive and verify on device. |
| Upgrade firmware | `npu-smi upgrade -t item -i dev_id -f file_path` | `file_path` supports only letters, digits, underscore, dot, slash, and hyphen. VRD firmware upgrade does not support concurrent upgrade. Do not repeat VRD upgrade while an upgrade is running. After VRD upgrade, run `reboot`, wait at least 60 seconds, then perform DC power off/on and wait for NPU link-up. |

Example package placeholder:

```bash
npu-smi upgrade -t vrd -i 0 -f Atlas-A5-hdk-vrd-firmware_xxx.hpm
```

### TLS Certificate Management

| Purpose | Command template | Rules / constraints |
| --- | --- | --- |
| Get TLS CSR | `npu-smi info -t tls-csr-get -i dev_id` | Gets the CSR file for the TLS certificate. |
| Import TLS certificate | `npu-smi set -t tls-cert -i dev_id -f "TLS证书 CA根证书 二级CA证书"` | File argument is a quoted string containing the TLS certificate, root CA certificate, and intermediate CA certificate. |
| Query TLS certificate info | `npu-smi info -t tls-cert -i dev_id` | Query certificate information. |
| Query expiration threshold | `npu-smi info -t tls-cert-period -i dev_id` | Query configured TLS expiration alarm threshold. |
| Set expiration threshold | `npu-smi set -t tls-cert-period -i dev_id -s period` | `period` range is `7..180` days; default `90`. |
| Restore default threshold | `npu-smi clear -t tls-cert-period -i dev_id` | Restores default certificate expiration threshold. |

KMSAgent prerequisite certificate setup from `j013`:

```bash
cd /var/kmsagentd
openssl genrsa -out root_ca.key 4096
openssl req -new -key root_ca.key -subj "/CN=Root CA" -out root_ca.csr
openssl x509 -req -in root_ca.csr -signkey root_ca.key -days 3650 -extfile openssl.cnf -extensions v3_ca -set_serial 1591942200000 -out root_ca.pem
```

If the driver was installed under a non-default path, replace `/usr/local/Ascend` in KMSAgent setup commands with the actual install path.

## Script-Generation Checklist

1. Discover devices first with `npu-smi info -l`; use returned NPU ID as `dev_id`.
2. Gate destructive commands behind explicit maintenance phases:
   - reset: `npu-smi set -t reset`
   - ECC clear: `npu-smi set -t clear-ecc-info` or `npu-smi clear -t ecc-info`
   - firmware upgrade: `npu-smi upgrade -t`
   - certificate import/threshold changes: `npu-smi set -t tls-*`
3. Do not run `npu-smi` commands in parallel.
4. Do not rely on example output formatting as stable; parse by labels defensively and keep raw logs.
5. For watch/statistics commands, treat `0` AI Core/Aivector usage under profiling as unsupported/meaningless, not as real utilization.
6. Validate absolute paths and free space before `sys-log-dump`.
7. Validate firmware file paths against the allowed character set before invoking upgrade.
8. Stop NPU workloads before reset and plan host reboot/power-cycle steps after VRD upgrade.
