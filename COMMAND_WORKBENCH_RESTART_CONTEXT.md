## 2. 最终目标

构建一个通用本地工具/插件，用于把测试用例文本转成一个可审阅、可调试、可保存的 **single 用例脚本**。

该工具应服务于如下场景：

1. 输入一段测试用例文本。
2. 自动识别出来
   - 预制条件。
   - 测试步骤。
   - 预期结果(并非每一个测试步骤或者预制条件存在预期结果，可以留空)
3. 根据用例文本
   - 针对预制条件和测试步骤生成可执行的bash命令，从用例文本中提取或整理可执行 bash 命令；如果原文没有明确命令，只能生成“参考命令/待确认命令”这些bash命令是要在一个session中执行(上下文会互相影响)
   - 大部分用例文本已经融合着自然语言把要跑什么脚本包含了，所以更多是在本地检索相关文档，确认命令的返回是什么格式，然后根据格式提取关键字，和预期结果对比
   - 预期结果也是脚本，需要可执行，可输出pass or nopass&why 
   - 每一个step都可以出发‘执行’，但是因为有上下文影响，所以‘执行’意味着‘从最开始的预制条件开始全量执行到此’
4. 工具启动本地 server + HTML UI。
   - UI建议以预制条件、测试步骤顺序从上至下单列排序，对于可以提供明确进行结果判断的在右侧对齐地方补充校验脚本；对于暂时无法提供的可以留空
   - UI更右侧提供console台，右侧上部体现每一次调试执行的结果；下侧提供与模型或者cli的交互记录，前期可以考虑仅使用简要说明，比如“正在进行文档分割”“正在进行预制条件确认”“加载sxxxskills中”
5. 使用LLM API或者CLI启动
   - 需要考虑多agent的方式，比如一个agent对文本进行切割，分别切割出对应的预制条件/测试步骤 vs 预期结果的组合序列
   - 绝对禁止捏造指令，所有的指令必须有源头：对于原文中出现的指令，主要工作是从文本中查询结果格式，方便快速获取关键字；对于原文中没有出现的指令，给出参考命令，要允许修改，确认修改后可以提示是否作为资产存入memory或者skill,供后续参考
   - 需要构建合理的本地信息库，或许是skill或者memory等，因为会有大批量的指令文档
6. 最终保存为一个完整独立脚本
   - 脚本执行后是按照 预期结果满足/不满足&why,测试步骤pass/nopass&why的方式执行
   - 脚本执行采用快速失败模式，一旦不符合预期立马中之
   - 最终可以保存全量的操作输入输出

## 3. 核心原则

### 3.1 输入文本是主来源

本项目的本质不是“根据自然语言自由生成测试脚本”，而是：

> 从已经包含命令、步骤、预期结果的用例文本中，做结构化提取、证据增强、断言生成和最终脚本拼装。

因此：

- 原文中的命令优先级最高。
- 原文中的步骤顺序优先级最高。
- 原文中的预期结果优先级最高。
- 文档只能辅助理解和修正，不能凭空创造命令。

### 3.2 禁止本地关键词/白名单召回参与主流程

已明确否定：

- 不要用简单关键词匹配来生成命令。
- 实际会有成千上万种工具、脚本、命令、管道、登录动作。
- 命令提取不能依赖固定工具名白名单。

正确做法：

- 使用模型/CLI 做结构化提取。
- 对“已经从输入中提取出的命令”再做文档证据增强。
- 文档不参与命令召回，只参与字段解释、输出说明、断言依据、修正建议。

### 3.3 文档是证据库，不是生成源

文档集合包括但不限于：

- `npu-smi` 说明。
- `hccntool` 说明。
- 未来其他大量自定义工具/脚本说明。

文档用途：

- 查命令格式。
- 查参数含义。
- 查输出字段。
- 查返回值说明。
- 查字段是否适合用于断言。
- 当原始命令疑似错误时，给出有证据的修正建议。

文档不应该：

- 从文档中自由构造用例没有写出的脚本。

## 4. 最终产物形态

### 4.1 脚本固定输出

最终脚本应包含这些固定输出，不在 UI 里作为可编辑字段出现：

1. 预制条件阶段完成：

```text
预制条件检查完成: PASS
```

或：

```text
预制条件检查完成: NO PASS
```

2. 用例执行阶段完成：

```text
用例执行完成: PASS
```

或：

```text
用例执行完成: NO PASS
```

3. 失败时输出样本对比：

```text
错误样本: ...
期望样本: ...
```

4. 最后一行/最终结果：

```text
PASS
```

或：

```text
NO PASS
```

退出码：

- PASS -> `0`
- NO PASS -> 非 0，建议 `1`

## 5. UI 目标

UI 是调试和人工确认工具，不是最终脚本定义本身。

### 5.1 UI 应展示

- 原始用例文本。
- 预制条件/测试步骤的有序序列，每步生成的命令 以及 原文说明
- 每步验证脚本 以及 原文说明。
- 文档证据(非必须，考虑后期实现)。
- 配置项目：
  - 被测试主机IP、密码等必要信息，录入后立刻启动连同性测试
  - skill/memory列表，可以以新页面或者弹窗方式呈现，这不是主流程，但是前期很多比如’判断版本‘等难以书写的脚本，要提供’cat xx/xx/version‘的映射，这时候可以留空并提示用户输入，pass后提示是否需要落盘为skill或者文档等技能
- 充分利用页面宽度


### 5.2 UI 执行语义

命令之间可能存在上下文依赖，例如：

- 环境变量。
- 当前目录。
- 登录状态。
- 上一步命令产生的目录或变量。

因此不能每个命令独立执行。

需要支持：

> 执行到此位置

即：从当前 session 开始，顺序执行到某个 step。

Output 只需要一个主输出区域，并允许展开查看所有输出(历史的可以折叠起来)。

### 5.3 UI 不应展示为可编辑字段的内容

以下内容属于最终脚本固定模板，不应在页面上作为人工修改项：

- `预制条件检查完成: PASS/NO PASS`
- `用例执行完成: PASS/NO PASS`
- `错误样本`
- `期望样本`
- 最终 `PASS/NO PASS`

## 6. 远程执行

远程执行是基本能力，因为被测机器不一定在本地。

需要支持：

- 目标 IP。
- 用户名。
- 密码或密钥。
- 端口。
- root 用户登录。
- host 侧命令。
- device 侧命令。
- host/device 上下文切换。

暂不做：

- 权限管理。
- 多用户隔离。
- 数据库。
- 并发调度。

但需要有本地持久化路径，保存：

- 固化脚本。

## 7. LLM API/CLI 设计原则

### 7.1 需要 CLI/LLM planner

简单本地规则不足以处理真实复杂用例。

因此主流程应依赖：

- Claude CLI。
- Codex CLI。
- 或其他本地 LLM/API wrapper。

### 7.2 需要对本地cli/LLM API进行判断

允许用户配置LLM API或者本地CLI,要进行联通性验证
